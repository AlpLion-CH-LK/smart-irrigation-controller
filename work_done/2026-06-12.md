# Work Done — 2026-06-12

## Goal
Fix the core training blocker found in the last run: the simulated plant dies during
Germination (Stage 0) in **every** episode (`death_rate=1.0`, `survival_rate=0.0`,
`final_stage=0`), so the PPO agent never experiences 96% of the 150-day season.

---

## Analysis

Reviewed the codebase and the 2026-05-28 wandb run. Confirmed:
- Episodes ended after ~140–293 of 3,600 steps, always in Stage 0 (Germination).
- Reward dominated by the terminal death penalty (`-100` for stage 0 at the time).
- Secondary issue: weather was sampled hour-by-hour **independently at random** from
  20 years of NASA POWER data — no day-to-day persistence, so "dry spells" weren't
  realistic.

## Fix B — Continuous real-history weather sampling

### `src/irrigation/weather/weather_data.py` (rewritten)
- Old: indexed records by `(hour, month)` / `(hour, month, year)` buckets, each hour
  sampled an unrelated random historical record.
- New: loads the CSV, sorts by datetime, builds **one continuous chronological list**
  of `WeatherRecord` (+ parallel `_months` list).
- New constants: `YALA_MONTHS = [1, 2, 3]`, `MAHA_MONTHS = [8, 9]` (replace old
  `FAVORABLE_MONTHS` / `ALL_MONTHS`).
- New methods:
  - `random_start_index(episode_hours, allowed_months=None)` — picks a random start
    index such that `record_at(i + h)` is valid for `h in [0, episode_hours)`. No
    wrap-around — excludes near-end-of-dataset starts.
  - `record_at(index)` — returns the `WeatherRecord` at a continuous timeline index.
- Removed: `sample()`, `sample_for_year()`, `get_years_for_month()`, `_year_index`,
  `_month_years`.

### `src/irrigation/sensors/historical.py` (rewritten)
- Removed the old `_pick_start()`, `_advance_month_if_needed()`, and per-`(hour, month,
  year)` bucket sampling / year-wraparound bookkeeping.
- New `HistoricalWeatherSensor(loader, soil_sensor, zone, training_phase=1,
  episode_hours=3600)`:
  - `_allowed_start_months()` — Phase 1 → `YALA_MONTHS`, Phase 2 → `MAHA_MONTHS`,
    Phase 3 → `None` (any month).
  - `reset()` — picks `self._index = loader.random_start_index(episode_hours=...,
    allowed_months=self._allowed_start_months())`, then resets the soil sensor.
  - `read()` — fetches `record_at(self._index)`, converts ET₀ → drain rate, applies
    rain, reads soil, increments `self._index += 1`, returns `SensorReading`.

**Result:** each episode now replays a real, contiguous slice of history — gradual
drying trends, multi-day rain events, real diurnal cycles — instead of "weather
whiplash" from independent hourly sampling.

## Fix C — Death-disabled warm-up phase

Initial attempt (loosening `germination` death thresholds in `config.yaml`) was
reverted: surviving the Stage-0 window is impossible for a near-random policy
regardless of small threshold changes, and the agronomic values are domain-correct and
shouldn't need to be loosened. Instead, implemented a **death-disabled warm-up**:
agronomic thresholds stay untouched, but episode termination on death is suppressed
for the first N training steps so the agent can still experience the full season while
its policy is close to random.

### `config.yaml`
- Reverted the germination threshold edits back to original values
  (`dry_hours_limit: 2`, `wet_hours_limit: 4`, `vitality_drain: 0.20`).
- Added new `training.death_warmup_steps: 20000` with explanatory comment.

### `src/irrigation/rl/gym_env.py`
- `reset()`: narrowed randomized initial soil moisture from `uniform(40, 80)` to
  `uniform(50, 65)`, so episodes don't start right next to a death boundary.
- Wired `episode_hours=self._max_steps` into the `HistoricalWeatherSensor` constructor
  (needed by Fix B).
- `__init__`: added `death_warmup_steps: int = 0` parameter, plus a persistent
  `self._total_steps = 0` counter (not reset per episode).
- `step()`: if the plant dies, the death penalty is still added to the reward; while
  `_total_steps <= death_warmup_steps`, the episode does **not** terminate —
  `PlantVitalityTracker.reset()` is called and the simulation continues to the full
  3,600 steps. After the warm-up window, normal death-terminates-episode behaviour
  applies.

### `scripts/train.py`
- Reads `death_warmup_steps` from config and passes it to Phase 1 env kwargs only
  (`{"zone": zone, "training_phase": 1, "death_warmup_steps": death_warmup_steps}`).
  Phase 2/3 don't get the warm-up — by then the policy should have learned the basics.

## Water-calibration recalibration

Diagnostic runs after Fix B + Fix C confirmed the warm-up mechanism works mechanically
(episodes ran the full 3,600 steps, dynamic stage progressed to Stage 3), but revealed
the agent was applying **10–25x too much water** (112–127 L/day vs. 5–12.7 L/day
needed), causing near-constant waterlogging during warm-up episodes.

### `config.yaml`
- `zone.max_litres_per_event`: `15.0 → 2.0`
- `zone.emergency_min_litres`: `2.0 → 0.2`
- `crop.etc_litres_per_day` rescaled (~6.3–6.9x down) to match the new action→water
  mapping:
  - `stage_0`: `5.0 → 0.8`
  - `stage_1`: `8.6 → 1.3`
  - `stage_2`: `12.7 → 1.9`
  - `stage_3`: `12.2 → 1.8`
  - `stage_4`: `10.3 → 1.5`
- `reward.death_penalty.stage_0`: `-100.0 → -20.0`
